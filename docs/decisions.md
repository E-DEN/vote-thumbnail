# 設計判断メモ

設計上の判断根拠を記録する。UX・実装の詳細はコードを参照。

---

## YouTube API 方針

### ユーザーアクセス時に YouTube API を叩かない

```text
user -> DB
           ↑
        worker
           ↑
        YouTube
```

ユーザーリクエストに応じて YouTube API を叩くと quota 枯渇・レスポンス悪化・キャッシュ不能が起きる。
そのため取得は非同期バックグラウンド処理に限定し、ユーザーは常に DB を参照する。

### `search.list` を使わない

`search.list` は 100 units 消費するため、公開サービスでは quota が即死する。
チャンネル検索 UI は作らず、YouTube URL / @handle 入力のみ対応する。

### @handle 解決は HTML パース

`@handle` から channelId を得るために YouTube HTML から `"channelId":"UCxxxx"` を抽出する。
API 消費なし。DOM 変更に備えて実装は抽象化しておく。

## 評価アルゴリズム

### Glicko-2 採用

Elo を採用しない理由：

- K係数は不確実性の手動近似にすぎない
- 対戦数が少ないサムネの信頼度を DB 外のロジックで補う必要が生じる

TrueSkill を採用しない理由：

- 元々チーム戦向けの設計で2択投票には過剰
- Microsoft 特許が商用利用に制約を与えうる

Glicko-1 ではなく Glicko-2：

- RD（レーティング偏差）が低対戦数サムネの信頼度を原理的に表現する
- 安定性パラメータ σ により評価がぶれるサムネを検出できる
- Glicko-1 との実装コスト差は小さい

RD が高いサムネはランキング表示時に視覚的に薄くする（信頼度が低いことをそのまま表現できる）。

## 荒らし対策

### 生 IP を保存しない

Workers で取得した IP を `sha256(ip + yyyy-mm-dd + secret)` でハッシュ化して保存。
日付を混ぜることで同日重複判定は可能にしつつ、日跨ぎ追跡を不可にする。

### レートリミットの数値を表示しない

「あと3回」「1分後に解除」のような具体値は表示しない。理由：

- BOT が閾値探索しやすくなる
- VPN 切替タイミングを最適化される
- 制限条件に合わせた自動化が容易になる

抽象的な状態表示（活動量・クールダウン中など）のみを使う。

## Reaction Pin 採用経緯

当初は Attention Map として「クリック地点を直接ヒートマップ入力」する設計を検討した。
しかしいくつかの課題が見えたため、Reaction Pin 方式へ転換した。

### Attention Map (直接入力) の課題

- 「分析ツール感」が強く、ユーザーに「どこ押せばいいか」が直感的に伝わらない
- クリックの意味が曖昧（熱量表現なのか正確な1点入力なのか）
- 誤クリック時に undo / 確認 UI が必要になり UX が重くなる
- 連打・BOT 対策が複雑化しやすい

### Reaction Pin の利点

- 「好きな場所へ pin を置く」として直感的に理解できる
- 最新位置のみ保持（1人1pin）なので押し直しで自然に修正できる
- 1人1pin なのでスパム効率が低く、高度な対策が不要
- ヒートマップは pin の集積から自動生成できる（UX と統計生成が一致）

結論: ユーザー操作は Reaction Pin に統一し、Heatmap は閲覧専用の生成物として扱う。

## Reaction Pin データモデル

### pin レコード（サーバー側 DB）

| フィールド | 型 | 説明 |
| --- | --- | --- |
| `video_id` | string | YouTube 動画 ID |
| `session_id` | string | 匿名セッション識別子（Cookie 等） |
| `x` | float 0.0–1.0 | 画像内相対 X 座標 |
| `y` | float 0.0–1.0 | 画像内相対 Y 座標 |
| `updated_at` | datetime | 最終更新日時（UPSERT） |

座標は画像サイズ非依存の相対座標とする。
PC・スマホ・レスポンシブ等の表示差異を吸収するため。

## ヒートマップ生成

### 密度ベース描画

固定スコア制は採用しない。
画像ごとにユーザー数・クリック総数が大きく異なるため、
その画像内での相対密度として描画する。

描画時には「その画像内で最も密集した場所」を 100% として正規化する。

### KDE アプローチ

各 pin を中心に Gaussian blur を重ねることで密度分布を生成する（KDE に近い考え方）。
「このピクセルを押した」ではなく「この周辺へ関心が集まった」という傾向を滑らかに表現する。

### diminishing returns（入力重み逓減）

将来的に1ユーザーの連続入力に重みを付ける場合は、
1回目 → 強い寄与・2回目 → やや弱い のように log / sqrt 的な圧縮を適用する。
現状の「1人1pin」設計ではこの処理は不要。
